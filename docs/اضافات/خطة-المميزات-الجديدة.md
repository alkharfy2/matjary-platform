# خطة المميزات الجديدة — منصة متجري (Matjary Platform)

> **تاريخ الإنشاء**: 25 مارس 2026  
> **الهدف**: تجهيز المنصة للإطلاق التجاري وتحقيق ميزة تنافسية في السوق المصري والعربي  
> **المرجعية**: تحليل المنافسين (Easy Orders, YouCan, Salla, Zid, ExpandCart)

---

## نبذة عامة

منصة **متجري** أنهت البنية التحتية الأساسية بنجاح: نظام Multi-Tenant، لوحة تحكم التاجر، واجهة المتجر، نظام الدفع (Kashier)، نظام الاشتراكات والمحفظة، ولوحة الإدارة العليا (Super Admin).

لكن الفجوة بين **"منصة تعمل تقنياً"** و **"منصة جاهزة للسوق"** كبيرة. التجار في مصر عندهم احتياجات محددة جداً:

1. **كل التجار بيعملوا إعلانات** — بدون Facebook Pixel و Conversion API مفيش تاجر هيشترك
2. **مشكلة الطلبات الوهمية** — أكبر ألم لتجار COD في مصر، ومفيش منصة حلّتها بشكل ذكي
3. **التجار عايزين سرعة** — الوقت = فلوس، أي أداة AI تسرّع عملهم هتكون سبب اشتراكهم
4. **التسويق والبيع** — Upsell, Cross-sell, Abandoned Cart هي اللي بتفرق بين متجر بيبيع ومتجر واقف

الخطة مقسّمة لـ **4 مراحل** حسب الأولوية، من الأساسي اللي بدونه مينفعش نطلق، لحد المميزات المتقدمة اللي هتبني عليها الحملات التسويقية.

### 1. Facebook Pixel + Conversion API

**الأولوية**: P0 — حرجة  
**السبب**: 95% من التجار المصريين بيعملوا إعلانات على Facebook/Instagram. بدون Pixel = بدون إعلانات = بدون تجار.

#### المطلوب:

**إعدادات التاجر (Dashboard → Settings)**:
- حقل `Facebook Pixel ID` في إعدادات المتجر
- حقل `Facebook Conversion API Access Token`
- حقل `Test Event Code` (للاختبار)

**الأحداث المطلوبة (Client-side Pixel)**:
| الحدث | المكان | البيانات |
|-------|--------|----------|
| `PageView` | كل صفحة في المتجر | — |
| `ViewContent` | صفحة المنتج | product_id, name, price, category, currency |
| `AddToCart` | زر إضافة للسلة | product_id, name, price, quantity, currency |
| `InitiateCheckout` | صفحة Checkout | items, total, currency |
| `Purchase` | صفحة نجاح الطلب | order_id, total, items, currency |
| `Search` | صفحة نتائج البحث | search_string |

**Conversion API (Server-side)**:
- إرسال نفس الأحداث من السيرفر مباشرة لـ Facebook
- مطلوب لأن iOS 14+ بيحجب الـ client-side pixel
- الأحداث: `Purchase`, `InitiateCheckout`, `AddToCart` على الأقل
- يتبعت مع: `event_id` (لمنع التكرار), `user_data` (hashed email/phone), `custom_data`

**التنفيذ في الكود**:
- مكون `<PixelScript />` يتحقن في `store/layout.tsx` لو التاجر حاطط Pixel ID
- Server Action أو API Route لـ Conversion API events
- Utility function: `trackPixelEvent(eventName, data)` — client-side
- Utility function: `sendConversionEvent(eventName, data, userData)` — server-side

**أعمدة قاعدة البيانات**:
```
stores.settings (JSONB) → إضافة:
  - facebook_pixel_id: text
  - facebook_conversion_api_token: text
  - facebook_test_event_code: text
```

### 4. WhatsApp Integration

**الأولوية**: P0  
**السبب**: WhatsApp هو وسيلة التواصل الأولى في مصر. كل متجر لازم يكون عنده طريقة سهلة للتواصل مع العملاء.

#### المطلوب:

**أ) زر WhatsApp عائم (Floating Button)**:
- يظهر في كل صفحات المتجر (أسفل يمين/يسار)
- يفتح رسالة WhatsApp مكتوب فيها: "مرحباً، أنا من متجر [اسم المتجر]..."
- التاجر يقدر يخصّص:
  - رقم الواتساب
  - نص الرسالة الافتراضية
  - إظهار/إخفاء الزر
  - موضع الزر (يمين/يسار)

**ب) زر "اطلب عبر WhatsApp" في صفحة المنتج (اختياري)**:
- يرسل رسالة فيها اسم المنتج + الرابط + السعر
- التاجر يفعّلها/يلغيها من الإعدادات

**ج) إشعارات WhatsApp للتاجر (مرحلة لاحقة)**:
- إشعار عند وصول طلب جديد
- يتطلب WhatsApp Business API (أو خدمة وسيطة مثل Twilio/360dialog)

**أعمدة قاعدة البيانات**:
```
stores.settings (JSONB) → إضافة:
  - whatsapp_floating_enabled: boolean (default true)
  - whatsapp_floating_position: 'left' | 'right' (default 'left')
  - whatsapp_default_message: text
  - whatsapp_order_button_enabled: boolean (default false)
```

> ملاحظة: رقم الواتساب موجود فعلاً في `stores.whatsapp`

### 5. Transactional Emails (إيميلات تلقائية)

**الأولوية**: P0  
**السبب**: العميل اللي يطلب ومايوصلوش إيميل تأكيد يحس إن المتجر "مش محترف". وده بيقلل الثقة.

#### المطلوب:

**الإيميلات الأساسية**:
| الإيميل | المُستقبل | التوقيت |
|---------|----------|---------|
| تأكيد الطلب | العميل | فوراً بعد إنشاء الطلب |
| تأكيد الدفع | العميل | بعد نجاح الدفع الإلكتروني |
| تحديث الشحن | العميل | عند تغيير حالة الطلب لـ "shipped" |
| تأكيد التوصيل | العميل | عند تغيير الحالة لـ "delivered" |
| طلب جديد | التاجر | فوراً بعد إنشاء أي طلب |

**التنفيذ**:
- استخدام خدمة مثل Resend أو Brevo (مجاني حتى 300 إيميل/يوم)
- Template إيميل HTML بسيط بشعار المتجر وألوانه
- From: `noreply@matjary.com` أو `store-name@matjary.com`
- **لا يحتاج SMTP Setup من التاجر** — المنصة بتبعت نيابة عنه

**أعمدة قاعدة البيانات**:
```
stores.settings (JSONB) → إضافة:
  - email_notifications_enabled: boolean (default true)
  - merchant_email_on_new_order: boolean (default true)
```

**الأولوية**: P0  
**السبب**: كل تاجر محتاج يطبع فاتورة مع كل طلب + بوليصة شحن لشركة الشحن.

- زر "طباعة بوليصة الشحن"

  - اسم المتجر (المُرسل)
  - اسم العميل + العنوان + رقم الموبايل
  - رقم الطلب
  - المحافظة + المنطقة

- CSS مخصص لـ `@media print`
- أو استخدام مكتبة PDF (مثل `@react-pdf/renderer`)

### 9. حاجب الطلبات الوهمية (Fake Order Blocker)

**الأولوية**: P1 — ميزة تنافسية أساسية  
**السبب**: أكبر مشكلة لتجار COD في مصر. العميل يطلب ويرفض الاستلام، والتاجر يدفع شحن مرتين (ذهاب + رجوع). **ولا منصة عربية حلّت المشكلة دي بشكل ذكي.**

#### الفكرة:

**نظام تقييم العملاء التلقائي** على مستوى المنصة بالكامل:

```
عميل يطلب بالرقم 01012345678
    │
    ├── أول طلب → "عميل جديد" (neutral)
    │
    ├── استلم الطلب → نقاط ثقة +1
    │
    ├── رفض الاستلام → نقاط ثقة -2
    │   └── تاني مرة رفض → "عميل مشكوك فيه" ⚠️
    │   └── تالت مرة → "محظور" 🚫
    │
    └── عند أي طلب جديد في أي متجر على المنصة:
        └── النظام يعرض للتاجر: "العميل ده طلب 5 مرات / رفض 3 مرات / معدل الاستلام 40%"
```

#### المطلوب:

**أ) جدول عملاء المنصة (Platform-level)**:
```sql
platform_customers
  - id: UUID
  - phone: text (UNIQUE, hashed أو مشفّر)
  - total_orders: integer (عبر كل المتاجر)
  - completed_orders: integer
  - rejected_orders: integer
  - trust_score: decimal
  - is_blocked: boolean
  - first_seen_at: timestamp
  - last_order_at: timestamp
```

**ب) منطق عند إنشاء الطلب (Checkout)**:
1. ابحث عن رقم العميل في `platform_customers`
2. لو `is_blocked = true` → ارفض الطلب: "رقمك محظور من الطلب"
3. لو `trust_score` منخفض → اسمح بالطلب بس بلّغ التاجر بالتحذير

**ج) لوحة تحكم التاجر — عرض التحذيرات**:
- في صفحة تفاصيل الطلب: شارة "عميل موثوق ✅" أو "عميل مشكوك فيه ⚠️" أو "عميل عالي الخطورة 🚫"
- نسبة الاستلام + عدد الطلبات
- التاجر يقدر يحظر الرقم يدوياً (على مستوى متجره)

**د) تحديث تلقائي عند تغيير حالة الطلب**:
- `delivered` → completed_orders +1
- `cancelled` (بسبب رفض الاستلام) → rejected_orders +1
- إعادة حساب `trust_score`

**هـ) إعدادات التاجر**:
- تفعيل/تعطيل حاجب الطلبات الوهمية
- الحد الأدنى لنسبة الثقة (مثلاً: رفض الطلبات تلقائياً لو trust_score < 30%)
- السماح بحظر أرقام يدوياً

**الميزة التنافسية**: 
> **Shared Blacklist** — لما عميل يرفض استلام طلب في متجر A، كل المتاجر على المنصة هتعرف.  
> Easy Orders مش عندها ده. ولا Salla ولا YouCan ولا أي منصة تانية.

### 10. Abandoned Cart Recovery (استرجاع السلات المتروكة)

**الأولوية**: P1  
**السبب**: 70% من السلات بتتسيب بدون إتمام الشراء. استرجاع حتى 10% منهم = زيادة مبيعات كبيرة.

#### المطلوب:

**أ) تتبع السلات المتروكة**:
- لما عميل يوصل لصفحة Checkout ويدخل بياناته (اسمه + رقمه) بس ميكملش → نحفظ السلة كـ "متروكة"

**جدول قاعدة بيانات**:
```sql
store_abandoned_carts
  - id: UUID
  - store_id: UUID (FK → stores)
  - customer_name: text
  - customer_phone: text
  - customer_email: text (nullable)
  - items: JSONB (المنتجات اللي في السلة)
  - subtotal: decimal
  - recovery_status: text ('pending' | 'sent' | 'recovered' | 'expired')
  - recovery_sent_at: timestamp (nullable)
  - recovered_order_id: UUID (nullable, FK → store_orders)
  - created_at: timestamp
  - expires_at: timestamp (مثلاً بعد 7 أيام)
```

**ب) إشعار الاسترجاع**:
- بعد ساعة من الترك → إرسال رسالة WhatsApp أو SMS:
  > "مرحباً [الاسم]، سلتك في [اسم المتجر] لسه في انتظارك! كمّل طلبك من هنا: [رابط]"
- بعد 24 ساعة → إيميل تذكيري (لو متاح)
- الرابط يوصّل لصفحة checkout مع استعادة السلة تلقائياً

**ج) لوحة تحكم التاجر**:
- صفحة "السلات المتروكة" بقائمة كل السلات
- فلترة: معلقة / تم الإرسال / تم الاسترجاع / منتهية
- معدل الاسترجاع (recovery rate) كإحصائية
- زر "إرسال رسالة تذكير" يدوي

**د) إعدادات التاجر**:
- تفعيل/تعطيل الاسترجاع التلقائي
- وقت الانتظار قبل الإرسال (30 دقيقة / ساعة / ساعتين)
- تخصيص نص الرسالة
- اختيار قناة الإرسال (WhatsApp / SMS / Email)

### 11. نظام Upsell (البيع الأعلى)

**الأولوية**: P1  
**السبب**: بيزود متوسط قيمة الطلب (AOV) بنسبة 10-30%.

#### المطلوب:

**One-Click Upsell بعد الشراء**:
- بعد ما العميل يكمل الطلب (صفحة نجاح الطلب)، يظهرله عرض خاص:
  > "أضف [منتج X] لطلبك بخصم 20% — بضغطة واحدة!"
- العميل يضغط "أضف" → المنتج ينضاف لنفس الطلب بدون إعادة الدفع (لو COD) أو بـ charge إضافي (لو Kashier)

**التاجر يحدد**:
- أي منتج يظهر كـ upsell لأي منتج
- نسبة الخصم على الـ upsell
- الحد الأقصى للـ upsell offers في كل طلب

**جدول قاعدة بيانات**:
```sql
store_upsell_rules
  - id: UUID
  - store_id: UUID (FK → stores)
  - trigger_product_id: UUID (nullable — لو null يعني لأي منتج)
  - offer_product_id: UUID (FK → store_products)
  - discount_type: text ('percentage' | 'fixed')
  - discount_value: decimal
  - is_active: boolean
  - sort_order: integer
  - created_at: timestamp
```

### 12. نظام Cross-sell (البيع المكمّل)

**الأولوية**: P1  
**السبب**: يقترح منتجات مكملة تزود قيمة السلة.

#### المطلوب:

**عرض منتجات مكملة في صفحة السلة والمنتج**:
- في صفحة المنتج: قسم "منتجات ذات صلة" أو "عملاء اشتروا كمان..."
- في صفحة السلة: قسم "كمّل مشترياتك"

**التحديد**:
- **يدوي**: التاجر يربط منتجات ببعض (مثلاً: كفر موبايل مع الموبايل)
- **تلقائي (مستقبلاً)**: بناءً على بيانات الشراء الفعلية

**جدول قاعدة بيانات**:
```sql
store_product_relations
  - id: UUID
  - store_id: UUID (FK → stores)
  - product_id: UUID (FK → store_products)
  - related_product_id: UUID (FK → store_products)
  - relation_type: text ('cross_sell' | 'upsell' | 'related')
  - sort_order: integer
  - created_at: timestamp
```

### 13. نظام Down-sell (عرض الإنقاذ)

**الأولوية**: P1  
**السبب**: آخر محاولة لإقناع العميل اللي هيسيب المتجر بدون شراء.

#### المطلوب:

**Exit-Intent Popup**:
- لما العميل يحرّك الماوس ناحية زر الإغلاق (أو يحاول يسيب الصفحة في الموبايل)
- يظهر popup:
  > "مستنيك! خصم 10% على طلبك لو كمّلت دلوقتي 🎉"  
  > كود الخصم: SAVE10  
  > [كمّل الطلب]

**التاجر يحدد**:
- تفعيل/تعطيل
- نص الرسالة
- كود الخصم (يرتبط بنظام الكوبونات الموجود)
- الصفحات اللي يظهر فيها (المنتج / السلة / Checkout)
- عدد مرات الظهور (مرة واحدة لكل جلسة)

**التنفيذ**:
- مكون `<ExitIntentPopup />` — client-side
- يقرأ الإعدادات من StoreContext
- يسجل في localStorage إن الـ popup ظهر (يمنع تكراره)

**أعمدة قاعدة البيانات**:
```
stores.settings (JSONB) → إضافة:
  - exit_intent_enabled: boolean (default false)
  - exit_intent_message: text
  - exit_intent_coupon_code: text (nullable)
  - exit_intent_pages: text[] ('product' | 'cart' | 'checkout')
```

### 15. AI Store Builder — "ابني متجرك في 60 ثانية"

**الأولوية**: P2 — ميزة تسويقية ضخمة  
**السبب**: Easy Orders عندها "Easy AI" بس لصفحات الهبوط. أنتم هتعملوها للمتجر كله.

#### الفكرة:

```
التاجر بعد التسجيل:
  → "صف لنا متجرك في جملة واحدة"
  → "أنا بابيع ملابس رجالي في مصر"
  → AI يولّد:
     ✓ اسم متجر مقترح
     ✓ وصف المتجر
     ✓ تصنيفات مقترحة (قمصان، بناطيل، أحذية، إكسسوارات)
     ✓ ثيم وألوان مناسبة
     ✓ إعدادات مقترحة (عملة، لغة)
     ✓ 3-5 منتجات تجريبية كـ template
  → التاجر يراجع ويعدّل → "ابدأ"
```

#### المطلوب:

- Integration مع OpenAI API (GPT-4o-mini — رخيص وسريع)
- نموذج في صفحة Onboarding: حقل "صف متجرك"
- API Route: `POST /api/stores/ai-generate`
- الـ prompt يُوجّه الـ AI لإرجاع JSON بالبنية المحددة
- التاجر يقدر يقبل أو يعدل كل اقتراح

**أعمدة قاعدة البيانات**:
```
stores → إضافة:
  - ai_generated: boolean (default false)
```

**متغيرات بيئية**:
```env
OPENAI_API_KEY=sk-...
```

### 19. نظام Dropshipping المدمج

**الأولوية**: P3  
**السبب**: سوق ضخم في مصر — ناس كتير عايزة تبيع بدون مخزون.

#### المطلوب:

**أ) ربط مع موردين**:
- كتالوج موردين محليين (اختياري — في المستقبل)
- استيراد منتجات من روابط خارجية (AliExpress, Alibaba)

**ب) سير العمل**:
```
التاجر يختار منتج من كتالوج الموردين
    → يعدّل السعر والوصف
    → ينشره في متجره
    → عميل يطلب
    → الطلب يتحول للمورد تلقائياً
    → المورد يشحن للعميل
```

**ج) جدول قاعدة بيانات**:
```sql
store_supplier_products
  - id: UUID
  - store_id: UUID
  - supplier_name: text
  - supplier_product_url: text
  - supplier_price: decimal
  - retail_price: decimal (سعر البيع)
  - product_id: UUID (FK → store_products)
  - auto_order: boolean (طلب تلقائي للمورد)
```

### 20. نظام التسويق بالعمولة (Affiliate System)

**الأولوية**: P3

#### المطلوب:

- رابط إحالة فريد: `ahmed.matjary.com/?ref=AFFILIATE_CODE`

- طلب سحب الأرباح

**ج) جداول قاعدة البيانات**:
```sql
store_affiliates
  - id: UUID
  - store_id: UUID
  - name: text
  - email: text
  - phone: text
  - code: text (UNIQUE per store)
  - commission_rate: decimal
  - total_sales: decimal
  - total_commission: decimal
  - pending_commission: decimal
  - is_active: boolean
  - created_at: timestamp

store_affiliate_sales
  - id: UUID
  - store_id: UUID
  - affiliate_id: UUID (FK → store_affiliates)
  - order_id: UUID (FK → store_orders)
  - sale_amount: decimal
  - commission_amount: decimal
  - status: text ('pending' | 'approved' | 'paid')
  - created_at: timestamp
```
